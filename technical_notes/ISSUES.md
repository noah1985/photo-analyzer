# Recent Technical Issues

## Current State

当前 `photo` 预设已经不再使用历史的小型 VLM 路线，而是：

- `Salesforce/blip2-opt-2.7b`（BLIP-2）
- 工作方式：`caption -> taxonomy 映射`

所以本文件中的前几项问题，主要是在说明历史试验为何被放弃，而不是在描述当前 `photo` 的运行状态。

## Historical: 小型 VLM 结构化输出不稳定

现象：

- 模型经常返回模板化文本，例如 `short English description`
- 同一批照片里会出现大面积相似描述
- 结构化标签分组不稳定，很多结果最终退回纯规则层

影响：

- `photo` 预设虽然名义上更偏摄影语义，但真实输出不可靠
- 用户体验上会感觉“新模型比旧模型更差”

结论：

- 当前项目不再继续以小型结构化 VLM 作为主方案
- 现已收敛为传统 caption；`photo` / `git_large` 使用 BLIP-2（见 [MODELS.md](MODELS.md)）

## Historical: `photo` 模型下载后崩溃

现象：

- 模型文件下载完成后，推理阶段失败
- 一次明确问题是 `torchvision` 缺失导致处理器无法加载

影响：

- 用户看到“下载很久，结果直接报错”
- UI 会误以为模型已经可用

结论：

- 不再依赖那条模型链路
- 新方案改用当前环境可直接加载的传统 caption 模型

## 3. 下载进度事件实现有 bug

现象：

- 下载阶段回调参数签名错误
- 流式分析里曾出现下载时额外抛 `TypeError`

处理：

- 已修正进度事件发送逻辑
- Swift UI 现在可显示下载百分比、状态文本和初始化耗时

## 4. 错误提示 UI 交互不稳定

现象：

- 工具栏里的“出错”标签点开后直接展开文本
- 会撑坏布局，影响顶部工具栏可读性

处理：

- 改为弹出式错误详情
- 保留错误内容复制能力

## 5. Hugging Face 缓存策略不合理

现象：

- 模型即使已下载，本地每次新进程仍会先做远端检查
- 用户容易误以为“每次都在重新下载”

处理：

- 已改为：
  - 先查本地缓存
  - 本地有则直接使用
  - 本地无才联网下载

## Historical: `photo` 回退链路曾失效

现象：

- 一版实现中，`photo` 失败后退回 `balanced` 时又被参数错误卡住
- 结果表现成：
  - 新模型失败
  - 回退也失败
  - 最终只能靠纯规则标签

结论：

- 当前已不再保留这条复杂回退结构
- 直接回归更稳定的模型实现，减少分支复杂度

## Current: `photo` / `git_large` 仍未默认取代 `balanced`

现象：

- `photo` 已从实验性 VLM 与 GIT 收敛到 **BLIP-2 opt 2.7B**；`git_large` 为 **BLIP-2 opt 6.7B**（预设 key 未改以保持兼容）
- 真实样本上通常优于已移除的 GIT COCO，但是否稳定优于 `balanced` 仍因题材而异
- CPU 下 BLIP-2 两档明显更慢、更重

结论：

- 当前默认推荐顺序仍然是：`balanced -> photo（高质量补充）-> git_large（更重、建议 GPU）-> fast -> detailed`（详见 [MODELS.md](MODELS.md)）
- `photo` / `git_large` 是可试选项，不应在对外文案中描述成“确定更准”

## 7. taxonomy 规则扩展后容易出现泛化标签

现象：

- 加入大量摄影标签后，风格标签和场景标签容易泛化
- 典型表现：
  - `电影感`
  - `氛围感`
  - `近景`
  - `竖幅构图`
  经常被过度命中

当前状态：

- taxonomy 仍然是受控输出的核心
- 但模型输入质量仍然决定上限

结论：

- 标签体系不是主要瓶颈
- 主要瓶颈仍然是 caption 质量
