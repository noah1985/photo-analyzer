# Photo Analyzer 重做说明

## 目标

当前放弃 `local_gallery.html` 这条产品路线，不再继续投入浏览器本地页。

新的目标是：

- 用 **Swift** 写一个本地 App
- 界面保留现有 HTML 的产品感觉和交互结构
- 真正的图片分析继续由本地 **Python CLI** 负责
- Swift 负责本地 UI、目录选择、数量控制、任务触发、结果展示

这次不是在旧实现上修补，而是按新的产品边界重新组织：

- `Swift App` = 用户真正使用的产品界面
- `Python CLI` = 本地分析引擎

## 为什么放弃 HTML

直接双击打开的本地 HTML 无法稳定完成以下链路：

- 选择本地目录
- 调用本机 Python
- 等待分析完成
- 接收分析结果并展示

如果坚持用 HTML，要么退回前端自己分析，要么引入本地服务。两者都不符合当前目标。

因此这次明确放弃 HTML 作为主产品入口，改为本地 Swift App。

## Swift App 要做什么

Swift App 是新的主界面，第一版至少完成这些能力：

### 1. 主界面结构

界面风格参考当前 HTML：

- 顶部标题和说明
- 一个“选择目录”按钮
- 一个“测试照片数”输入框，最大 100
- 一个“开始分析”按钮
- 下方为 collection view 网格结果区

### 2. 用户操作流程

用户操作固定为：

1. 选择本地照片目录
2. 输入抽样数量
3. 点击开始分析
4. App 调用本地 Python CLI
5. 等待 CLI 返回结果
6. 在界面中展示图片、标签、总结

### 3. 结果展示

每张卡片至少展示：

- 图片缩略图
- 文件名
- 中文标签
- 一句中文总结

第一版不要求做复杂筛选、排序、编辑。

### 4. 错误处理

Swift App 需要明确处理：

- 目录未选择
- 数量非法
- Python 不可执行
- CLI 执行失败
- 返回结果为空
- 某些图片分析失败

错误要在界面里给出清晰提示，而不是静默失败。

## Python CLI 是干什么的

Python CLI 不再是给用户直接长期操作的主要入口，而是 Swift App 背后的本地分析工具。

它负责：

- 扫描目录中的图片
- 按指定数量抽样
- 调用本地图像分析逻辑
- 输出结构化结果

它不负责：

- 产品级 UI
- 手动浏览体验
- 本地交互式目录选择

也就是说，CLI 的定位是：

> 本地分析引擎和结果生产器

## CLI 当前应该承担的接口

为配合 Swift App，CLI 需要稳定提供一条机器可读入口。

推荐保留的接口是：

```bash
photo-analyzer sample-gallery <directory> --count <n> --output-dir <dir>
```

或者后续进一步收敛为类似：

```bash
photo-analyzer analyze-dir <directory> --count <n> --json
```

核心要求只有两个：

- Swift 可以稳定调用
- CLI 能稳定返回 JSON 结果

JSON 结果至少要包含：

- 图片路径
- 文件名
- 标签数组
- 总结
- 必要时附带错误信息

## 本次需求是什么

这次的真实需求已经明确为：

### 产品目标

做一个 **本地照片分析 App**

### 主入口

用 **Swift 原生界面**

### 分析方式

由 **本地 Python CLI** 提供分析能力

### 第一版交互

- 选择目录
- 选择测试照片数量
- 启动分析
- 展示图片和标签结果

### 第一版输出重点

- 中文标签
- 一句中文总结
- 可附带基础指标，但不是首要界面重点

### 当前不做

- 浏览器 HTML 主产品化
- 本地 web service
- 远程服务
- 云端分析
- 标签编辑 UI
- 复杂照片管理能力

## 下一步实现顺序

推荐按这个顺序推进：

1. 稳定 Python CLI 的 JSON 输出
2. 明确 Swift App 和 CLI 的调用协议
3. 搭建 Swift 主界面
4. 接通目录选择、数量输入和命令执行
5. 展示图片卡片和标签
6. 处理错误状态和空结果状态

## 当前结论

从现在开始，项目应按以下原则推进：

- HTML 不再是主产品
- Swift App 是新的前台
- Python CLI 是后台分析引擎
- 所有后续实现都围绕这两个角色展开
